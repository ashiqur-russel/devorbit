DEVORBIT • USER GUIDE

How Devorbit Works

A practical guide to the Devorbit user journey: teams & projects, pipelines, deployments (Vercel + VPS), server monitoring, and how to troubleshoot issues when data isn’t showing.

1) Big picture

  • Pipelines come from GitHub Actions/GitLab CI via API polling.
  • Deployments come from platform integrations (ex: Vercel) and CI reporting (ex: VPS/OVH).
  • Servers come from the Devorbit Agent running on your machine (CPU/RAM/disk/network + online/offline).

2) Roles & access control

Organizations are tenant boundaries. Teams are workspaces. Projects belong to teams.

  • SUPER_ADMIN: full control.
  • ADMIN: can manage, plus optional capabilities (canCreateTeams, canInstallAgent).
  • MEMBER: normal access.

3) User journey

  • Sign up or sign in (Email/GitHub)
  • Create or join an organization
  • Create a team (workspace)
  • Connect integrations (GitHub, Vercel)
  • Create projects and link repos / Vercel project IDs
  • Install agent on servers to see metrics

4) VPS deployments (OVH / any server)

VPS deployments are recorded via CI reporting. After a deploy succeeds, your CI calls: POST /api/v1/deployments/report with a deploy token.

Per project setup (Projects → Manage → VPS deployments):

  • Generate a deploy token (shown once)
  • Add repo secrets: DEVORBIT_PROJECT_ID + DEVORBIT_DEPLOY_TOKEN

5) Troubleshooting

  • Pipelines empty: check GitHub token permissions (Actions Read) + repo owner/name + runs exist.
  • Deployments only show Vercel: expected unless VPS reporting is configured (deploy token + workflow reporting).
  • deployments/report fails: wrong project id, wrong token, newline in secret, API unreachable, or proxy issues.
  • Server offline: agent not sending heartbeat or stale thresholds too aggressive.
Common “report” mistakes:
- DEVORBIT_PROJECT_ID must be the Project ID (24-hex), not the Team ID
- DEVORBIT_DEPLOY_TOKEN must not include a trailing newline

6) What to collect for support

  • Screenshot of the page (Pipelines/Deployments/Servers)
  • The project + team you expected
  • CI log lines from the failing step
  • API logs if the server is self-hosted
